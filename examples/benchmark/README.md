# Benchmark — ガイドライン実証ループ

今のプロセス（`guidelines/` + `prohibited.md` + 決定順）で実在の優良サイトと同等のものを作れるか検証し、及第点を割る箇所からガイドライン/手法を補強する。

## 方法論の修正（重要）

**第1回は採点が甘く、撤回した。** 当初は「ヒーロー1枚＋数セクション」のビルドを、スクショを一目見て自己申告で PASS にしていた。実物は ~8–18 セクションの全長ページで、これでは**実物の約1割**しか作れておらず比較にならない。

**修正版の方法論:**
1. 実物ページを**コード（HTML/CSS）として `refs/` に保存**（curl + `<base href>`）。これを"正解"とする。画像は外部依存で欠けるが、構造・本文・配色・書体は実物が入る。
2. 比較は**視覚でなく「実コード vs 生成コード」**で行う（→ エージェント並列で回せる）。
3. 各サイトに1エージェント: 実コードから**全セクションを blueprint 化**（`refs/blueprints/`）→ 決定順で**全長ページを生成** → 実コードと突き合わせ**完全性を含む厳密採点** → 欠落を改善 → 再生成。
4. エージェントが報告した**ガイドラインの欠落を集約して本体を改善**。

## 及第点（合格基準）— 各ビルドがすべて満たす

1. **完全性**: 実物の**全セクションを順序通り cover**（real N vs built M、スキップ＝不合格）← 最重要・最初に確認
2. 題材/ブランド意図が伝わる（bespoke 書体・配色を題材から導けている）
3. 視覚階層・可読性・余白リズムがプロ水準
4. 構造が題材に適合（生成テンプレ骨格でない）
5. prohibited 違反なし ＋ a11y 基本（contrast・focus・色非依存）
6. 「実在の出荷可能なサイト」として通用する（最重要）

> 思想が違うので完全一致は不要。"正解"（実サイト）と比べて及第点を超えればよい。

## 次ラウンドの方法論改善（2026-07 外部知見の取り込み）

評価・ベンチマーク手法の調査（→ `memo/research-ai-design-ecosystem-2026-07.md`）から、次ラウンドで採用する改善:

1. **セクション単位の採点**（WebGen-V, arXiv 2510.15306）: ページ全体の総評は欠陥を見落とす（セクション単位の欠陥検出 F1 0.78 vs 全ページ 0.46）。blueprint の各セクションに対し Text（内容正確性/配置/可読性）・Layout（重なり/整列/gap 一貫性）を個別採点し、**閾値未満のセクションだけ狙い撃ちで再生成**する。全ページ作り直しをしない。
2. **ルーブリックは Originality に重みを置く**（Anthropic ハーネス設計）: 4軸 = Design Quality（一貫した視覚的アイデンティティ）/ **Originality（テンプレ既定値でなく意図的な判断の痕跡 — 重点）** / Craft（spacing・階層・コントラスト）/ Completeness。Craft は元々強いので、採点コメントは Originality と Design Quality に厚く。**AI収束スタイル（prohibited「第2デフォルト」3類型）に落ちていないか**を Originality の必須確認項目にする。
3. **チェックリスト駆動の批評**（ArtifactsBench）: 評価エージェントに自由記述の総評をさせず、サイトごとに blueprint から**細粒度チェックリストを先に生成**させ、各項目に pass/fail＋根拠で答えさせる（自由総評は甘くなり、ドリフトする）。
4. **機械 lint を先に回す**: `tools/design-lint.py` を批評の前に実行し、機械判定可能な違反（transition:all・グリッド外 spacing・text-wrap 欠落等）を潰してから、人的判断が要る軸だけ批評に回す。
5. **ガイドライン改善の効果検証はブラインド対比較で**（Justin Wetch 方式）: 改善前後のガイドラインで同一題材を生成し、スクショを匿名化して判定者（別エージェント）に対比較させる。「改善した気がする」を統計に置き換える（先行事例: 28決着比較で 75% 勝率、p=0.0125）。

## 第2ラウンド結果（2026-07 — 上記方法論で全10サイトを再生成）

外部知見取り込み後のガイドライン（状態色導出・APCA・ダーク導出・text-wrap/tabular-nums/66ch・モーション実装床・文言規約・6マイクロステート・AI収束3類型・実装床）＋新lint（8チェック追加）で全10ビルドを再生成し、blueprint 完全性照合・lint・収束判定・レンダリング検証（spot 4件: 01/04/06/07 — 全長・横overflow 0 を確認）を通した。

**結果: 10/10 合格。lint 合計 0 error**（開始時 WARN 計 ~30 → 6。残6件は全て「実物 identity 由来」の根拠コメント付き: framer の真黒地/純白CTA、pool の実測光学値 114/29/130px、squarespace のアイコンボタン等padding、clarasight のブランドグラデ・章ラベル）。

主な修正: fourmula の等分3カラム（テンプレ型）→実物準拠の縦積み大ブロックへ構造修正／pool の5秒自動巡回タブ撤去（Calm 準拠）／cartesia・squarespace のタッチターゲット・見出しスキップ是正／norma の font-size 17段→11段統合・hover 導出方向修正／全ビルドに cap-height 補正・staggered reveal 1回集約・color-mix/oklch 状態色導出を統一適用。

**収束3類型判定**: 全10とも「デフォルト収束でなく題材由来」と判定（fourmula=黒地×朱は実物由来、cartesia=クリーム×セリフは実物 PP Kyoto 由来など、各ビルド冒頭に導出根拠コメントを記載）。この「該当しても導出理由を書けば可、書けなければ修正」という運用が判定を機械的にした。

**このラウンドが表面化させた欠落（次の改善候補）**:
1. lint がコメント内 hex を誤検知（→ **修正済み**: コメント除去プリパス追加）
2. クローン/既存ブランド継承タスクで「06 のウェイト両極端」等の推奨が実物 identity と衝突 — 「実測再現は identity 優先」のカーブアウトを 06/color-system に一文
3. アイコンオンリーボタンの等 padding は正当（非対称化するとグリフ偏心）— spacing-system/prohibited に例外明文化
4. 09-motion: ページロード演出（staggered reveal）の duration 目安が未定義（500ms は許容か）／marketing.md のマーキー許容との相互参照
5. lint に「identity 例外」を宣言する仕組み（lint-allow アノテーション）がなく、題材由来 WARN が恒久ノイズ化
6. 行長（measure）・hover 導出方向（コントラストを下げる mix）の lint 化余地
7. footer カラム見出しの heading レベル指針（h2→h4 スキップが頻発パターン）

## 発見した欠落 → ガイドライン改善（本ラウンドの収穫）

実コード対標＋完全性採点により、6体の報告が**一致して同じ欠落**を指摘。すべて本体に反映済み:

1. **全長/完全性の床が無い**（全員指摘・最大）→ `01-foundations`「**全長を満たす**」追加。ヒーロー1枚は形式上ガイドラインに適合できてしまう＝旧ビルドが自己 PASS した原因。実物を対標するなら全セクションを blueprint 化し欠落ゼロで cover、スキップ＝不合格、と明記。
2. **ダークテーマ配色が無い**（framer/norma/unmoth/fourmula/prime/squarespace が各自手作り）→ `values/color-system`に**ダーク地ランプ＋bespoke 導出手順＋on-accent（アクセント面に対して測る）**を追加。
3. **06 が「見出し専用書体を増やさない」で 01 と矛盾＋アプリ寸法**→ `06-typography`に**ブランドモード（ディスプレイ書体可）＋ディスプレイスケール `clamp(40,6vw,88)`**を追加し矛盾を解消。
4. **マーケ固有パターンが無い**（製品モック・写真ヒーロー・索引カタログ・configurator・マーキー）→ 新規 `components/marketing.md`。等分3カラムでなく「索引/カタログ様式」で密な列挙を扱う、を明記。
5. **「1アクセント」が広すぎ／ゼロアクセント不可**→ `color-system`カーブアウト: 1アクセントは*UIシグナル色*の規律（コンテンツ画像・ロゴ・作例は対象外）、モノクロ志向はゼロアクセント可。
6. **索引連番・三次タッチターゲットの例外が無い**→ `prohibited`: 編集サイトの章/作品番号 `001…` は実索引なので可。フッター等の三次リンクは WCAG 2.5.8 の 24px 下限まで可。

## 結論

第1回の甘い採点（ヒーロー1枚で自己PASS）を撤回し、**実コードを正解にした全長リビルド＋完全性採点**に切り替えた結果:

1. **10/10 が全長で再生成され、私が1つずつ描画検証**した。実物の identity・構造・実コンテンツを捉えており、及第点6項目を満たす出荷可能水準（当初の軽微2件も修正・再検証済み）。
2. より重要な収穫として、ベンチマークが**7件の具体的なガイドライン欠落**を表面化させ、全て本体に反映した（全長/完全性の床・ダーク配色・ブランド書体スケール・マーケパターン・アクセント規律・索引/タッチの例外）。

これにより、ガイドライン＋プロセスは**道具UIだけでなく、ブランド/マーケ/編集系の全長サイトも及第点で駆動できる**ことが、甘い目視でなく実コード対標で実証された。

軽微2件は修正・検証済み。**全10サイトがクリーンに及第点を突破**。改善後ガイドラインでの再生成（穴が塞がったかの再検証）は任意の追加ステップ。
