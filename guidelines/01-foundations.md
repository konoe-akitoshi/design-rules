UI を作る前の姿勢と順序。すべての章がこの章を前置きとして読む。

## 題材から構造を導く（A｜テンプレ骨格の禁止）

**思想**
ページの構造は、汎用テンプレートからでなく、**題材そのものの native な様式**から導く。

**その思想を使うための理論**
テンプレートを流用した画面が「どこかで見た」になる最大の原因は、配色でも題材でもなく**レイアウトの骨格（アーキタイプ）**。ナビ→ヒーロー(eyebrow＋大見出し＋2ボタン＋右図版)→等分3カラム特徴→中央引用→3カラムフッター、という"ランディングの定番骨格"は、中をいくら抑制しても「どこかで見た型」を発する。題材が違っても骨格が同じなら、同じ顔になる。
だから骨格を借りない。題材が持つ固有の**文書様式・道具・語彙**から構造を立ち上げる（計器メーカーなら実測図、辞書なら見出し語と語釈、時刻表なら罫と時分、台所道具なら工程）。これは Bauhaus「形は用途・素材から導く」と Tufte「data-ink」を**構造レベル**に適用したもの——全要素が情報を運び、装飾のためだけの構造単位を置かない。

**使える規則**
- 構造を決める前に「この題材は本来どんな様式（図面・目録・票・台帳・記事…）で表されるか」を1つ挙げ、そこから章立てを導く。
- 汎用ランディング骨格（eyebrow＋1語色付き大見出し＋塗り/線2ボタンのヒーロー、等分3カラム特徴、中央寄せ引用、装飾的な 01/02 番号）を既定で使わない。使うなら「この題材で本当にこの型か」を毎回問う。
- 各構造単位（節・帯・カラム・図）に「何の情報を運ぶか」を言えること。言えない単位は削る。
- ヒーローは"型"でなく**主張**：題材の最も特徴的なものを、それに相応しい形で最初に置く。

## 題材からブランドを導く（書体・配色）— ブランド/編集系のとき

**思想**
構造だけでなく、**ブランドの素材（書体・配色）も題材から導く**。道具のベースライン（system font ＋ `values/color-system` の slate+青）は、ブランドの identity ではない。

**その思想を使うための理論**
ブランド/マーケ/編集系サイトでは、**ディスプレイ書体と配色が identity の半分**を担う。system font と汎用 slate パレットのままだと、構造が題材から導けていても「中立＝どこかで見た」印象になる（例えば編集的な製品なら、writerly なセリフ＋温かい紙色＋低彩度の1アクセントが identity の核になりうる）。だから題材の世界から書体と色を1つずつ選ぶ。これは Bauhaus「素材から形を導く」の素材レベルの適用。

**使える規則**
- **書体**：ブランド系では題材に合うディスプレイ書体を1つ選ぶ（編集的ならセリフ、技術的ならグロテスク/モノ等）。Web フォント（Google Fonts 等）の使用可。本文はそれと噛み合う1書体。`tokens` の system font は道具/アプリのベースラインで、ブランドの既定ではない。
- **配色**：題材から **bespoke パレット**（地・インク・ヘアライン・1アクセント＋セマンティック）を作る。`values/color-system`（slate+青）は*道具*のベースラインで、ブランドにそのまま当てない。ただし役割構造（地/面/線/ink/muted/accent/semantic）・コントラスト・1アクセント＋セマンティックの**規律は共通**で守る。
- 純白(#FFF)・純黒(#000)を避け、わずかに色を寄せた地・インクにする（紙の温度・冷たさが identity になる）。
- 道具/ダッシュボード（B）では逆に、ベースライン（system font＋slate）のままで良い。ブランド派生は不要。

## 全長を満たす（ブランド/マーケのとき）

**思想**
ブランド/マーケのページはヒーローだけでは成立しない。題材の物語を**最後まで**運ぶ。

**その思想を使うための理論**
ランディングページは、ヒーローで主張を立てた後、それを裏づける筋（社会的証明・機能・実演・実例・料金・FAQ・最終CTA…）を順に展開して初めて成立する。ヒーロー1枚／1画面は「題材から構造を導く」原則に形式上は適合**できてしまう**が、主張を支える本体が無く未完成。抑制（07）は*セクション内の密度・装飾*を削るものであって、*セクション数*を削る免罪符ではない。

**使える規則**
- ブランド/マーケページは「題材の物語の全長」を作る。ヒーロー＋数セクション＋フッターで早じまいしない。
- 節は「読者が次に何を知る必要があるか」で並べる（主張→裏づけ→具体→行動）。題材が要求する節を落とさない。
- 抑制はセクション**内**の密度・装飾に効かせる。セクションを削って「ミニマル」と称さない。
- 機能など項目が多いときは、等分3カラムでなく**索引/カタログ様式**（見出し＋タグ列）で扱う（→ [components/marketing](./components/marketing.md)）。

## Experience Before Expression — 経験が先、表現が後

**思想**
ビジュアル表現はユーザーの経験を支えるためにある。表現を先に決めると、画面は整って見えてもシステムの理解や操作を助けないことがある。

**その思想を使うための理論**
認知の流れは `Experience → Structure → Interaction → Visual`。見た目を先に固定すると、後から情報構造を入れ替えられず、無理な辻褄合わせ（制約の逆流・手戻り）が起きる。だから「まず何を理解・判断・操作するか」を決め、見た目はその結果として導く。見た目を軽視するのではなく、**出発点にしない**。

**使える規則**
- 決定順（下の[決定順](#決定順-operating-procedure)）を守る。各段は前段に従属させる。
- 配色・装飾から設計を始めない。情報構造と導線が未定のままレイアウトに入らない。

## Pragmatism — 思想は作業を良くしなければ意味がない

**思想**
美学や理論を単独の正解として扱わない。使えること・試せること・改善できることを重視する。

**その思想を使うための理論**
デザインは「完成品」ではなく「作業を改善する検証可能な仮説」。だから判断の正当化は**権威（原則名）ではなく結果**（理解・作業速度・保守性の改善）で行う。原則同士が衝突したとき、または理論上の正しさと実際の作業改善が衝突したときは、**この場面で作業を改善する方**を採る。これがすべての規則に優先するメタ下限（`M-PRAGMATISM`）。

**使える規則**
- 各デザイン判断に「改善する具体的な作業効果」を1つ言えるようにする。言えないなら採用しない。
- 思想・原則を教条的に当てはめない。引用は実益がある時だけ。
- 一度で完成させず、観測して改善できる形にする。

## Stewardship — 作って終わりにしない

**思想**
良いデザインは運用され、観測され、修正され、長く使われる。価値は初期状態ではなく、変化に耐えるかで決まる。

**その思想を使うための理論**
サーバー・CMS・LLM・運用ツールでは、データ・例外・権限・保守者が時間とともに変化する。理想状態だけを設計すると、情報が増えた瞬間に崩れる。だから設計対象に**経年（データ増加・長文化）・例外（空/エラー/権限不足）・引き継ぎ（標準的で読める構造）**を含める。凝りすぎて変更困難な一点物より、拡張・引き継ぎ可能な標準構造を選ぶ。

**使える規則**
- レイアウトは 初期 / 空 / 大量(1000件) / 長文 のすべてで崩れないか確認する。
- 固定件数・固定文言に依存しない。
- 運用UIでは「誰が・いつ・なぜ変更したか」を追える（履歴・更新者）。
- （例外状態の具体は [08 states-resilience] で扱う。）

## Human-Centered — 人間を構造の内側に含める

**思想**
システムはインフラとデータだけで成立しない。人間の認知・判断・失敗・習熟を構造の一部として扱う。

**その思想を使うための理論**
ユーザーは仕様通り完璧に動く存在ではなく、状況を読み・迷い・確認し・間違える参加者（Situated Action）。頭の中で行う比較・計算・記憶は、UIが肩代わりすべき**認知労働コスト**。不可逆・重大な自動処理には人間の確認点（human-in-the-loop）を挟む。

**使える規則**
- ユーザーに暗算・記憶・複数画面の往復を強いない。比較対象は並置し、計算結果は事前表示する。
- 中断・再開・部分入力を許容する（一本道で例外を許さないフローにしない）。
- 誤操作・見落としからの回復導線を必ず用意する。
- 不可逆・重大な自動処理に承認ステップを置く。

## Systems Thinking（B｜システム/道具のレンズ）— デザインは翻訳である

> **適用条件**: 題材が複雑なシステム（サーバー・k8s・CMS・LLM・監視・運用ツール等）のときだけ被せる。ブランドページ・記事・印刷物には適用しない。**最上位概念ではない**。

**思想**
（システムを扱うとき）デザインは、複雑なシステムを人間が理解・操作・制御・保守できる形に翻訳する手段。

**その思想を使うための理論**
システムは `infrastructure + information + interface + human action` で成立し、UI はその間の「翻訳層」。設計対象は部品ではなく、要素間の**関係・循環・フィードバック・遅延**。運用法は2つ。(1) 画面が担う機能を6つ — 観測 / 関係読解 / 判断支援 / 介入点提示 / 結果フィードバック / 保守 — から特定し不足を補う。(2) レイヤー順 `Systems → IA → Interaction → Visual` を逆流させない（見た目の都合で構造を曲げない）。

**使える規則**
- 着手時に「何のシステムか / 誰が / 何を観測・判断・制御するか」を1文で言えるようにする。
- 画面の主目的を **観測 / 操作 / 判断支援** のどれか1つに定め、必要な要素が揃っているか確認する。
- 部品単体の最適化で全体の一貫性・認知負荷を悪化させない（部分最適の禁止）。
