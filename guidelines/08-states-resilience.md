# 08 — States & Resilience（状態と耐性）

理想状態だけで設計を完成させない。空・エラー・権限・経年に耐えてはじめて「使える」設計になる。
関心: **例外状態・フィードバック・回復**。決定順の段6。UI は一枚の完成絵ではなく、時間の中で変化し続ける仕組み。

> **層**: 主に **B**（システム/道具向け）。空・読込・エラー・権限などの動的状態を持つUIに適用し、静的なブランドページには当てない。
> ただし Stewardship の「データが大量・長文でも崩れない／固定値に依存しない」は **A** として全ジャンルに残る。

各項目は `思想 → その思想を使うための理論 → 使える規則` の3層。

---

## Stewardship — UI は完成絵ではなく時間に耐える仕組み

**思想**
設計は「作って終わり」ではなく、長く扱い・保守し・変化に耐えさせる責任。初期データの少なさで美しいだけの UI は弱い設計。

**その思想を使うための理論**
理想状態だけの設計が時間とともに壊れるのは、**データが設計者の前提を裏切るから**だ。ローンチ時は件数が少なく・文言が短く・例外がない。だが運用が進めば、リストは0件にも10万件にもなり、名前は折り返し、権限は分岐し、削除済み・期限切れ・移行中のデータが混ざる。固定件数を前提にしたグリッド、固定幅を前提にしたラベル、理想データを前提にしたレイアウトは、この変化のどこかで必ず崩れる。だから設計の検証単位を「理想の1枚」ではなく**初期/空/大量/長文/例外の集合**に広げる。崩れても意味を保って使い続けられる状態（Graceful Degradation）を最低ラインとし、加えて運用 UI では**誰が・いつ・なぜ変えたか**（Accountability）を追える構造を最初から組み込む。後付けは難しい。

**使える規則**
- 各画面を 初期 / 空（0件）/ 大量（数千件）/ 長文（最長ラベル・氏名・URL）で必ず検証する。理想データ1件で確認しない。
- 固定件数・固定文言に依存しない（「3件のカード」前提のレイアウト、改行しない前提の1行ラベルを禁止）。
- 崩れても意味を保つ（画像欠損→代替表示、未取得→既知の情報だけ表示、はみ出し→省略＋全文導線）。
- 運用・管理 UI には 更新者・更新日時・変更理由/履歴 を追える導線を置く（誰が・いつ・なぜを隠さない）。

## Cybernetics — 操作の結果が同じ文脈に閉じる

**思想**
UI は静的な表示面ではなく、観測→判断→制御→フィードバックの制御ループの一部。結果が次の入力に戻ってはじめてループが閉じる。

**その思想を使うための理論**
状態を扱う画面でユーザーがやっているのは「状態を見て、目標との差を判断し、介入し、結果を確認する」循環だ。このループが閉じない最大の原因が**逸脱（エラー・遅延・非同期）の握りつぶし**にある。操作したのに無反応に見える、処理が走っているのに画面が静止している、失敗したのに何が起きたか返らない——どれもユーザーから「結果の観測」を奪い、次の手を打てなくする。だからフィードバックは Error Correction（目標との差の検出と修正）まで含めて設計する。エラーは**検出（何がどう違うか）と復旧導線（どう直すか）を必ずセット**にし、別画面に飛ばして因果を切らない。処理中・遅延・非同期は「いま動いている」ことを可視化し、終わりの見える化（進捗・推定時間）で無反応との区別をつける。操作→結果を同じ文脈で閉じることが、ループを閉じる最小条件になる。

**使える規則**
- 操作の結果は、操作した場所から目を離さず確認できるようにする（別画面遷移で因果を切らない）。
- エラーは検出（原因）と復旧導線（次の一手）をセットで出す。エラーコードだけで放置しない。
- 非同期・時間のかかる処理は「処理中」を即時に可視化し、可能なら進捗・推定時間・完了通知を返す。
- 楽観的更新をする場合も、確定・失敗・再試行の最終結果を必ず同じ文脈に反映する。

## Human-Centered — 不確実性を扱い、失敗から回復させる

**思想**
ユーザーは仕様書通りに動かない。状況を読み、迷い、中断し、時に間違える。UI は理想フローだけでなく、例外・中断・回復を内側に含める。

**その思想を使うための理論**
人間の行為は固定の計画ではなくその場の状況に応じて変わる（Situated Action）。だから「全項目を一気に正しく入力し、最後まで中断せず完了する」理想フローだけを設計すると、現実の作業——途中で離席する・別タブを開く・あとで戻る・一部だけ入力する——がそのまま失敗になる。さらに危険なのが、**未確定・処理中の状態を確定済みに見せること**だ。保存前の入力を保存済みに、送信中を送信完了に、推定値を実測値に見せると、ユーザーは誤った判断を下し、その誤りに気づけない。これは不確実性を隠すことで認知を支えるどころか裏切る。だから状態の確からしさ（下書き/保存中/保存済み、推定/確定）を明示し、中断・再開・部分入力を許容し、誤操作には必ず回復導線（undo・復元・確認）を用意する。失敗を前提に置くことが人間中心設計の中身になる。

**使える規則**
- 未確定・処理中・推定値を確定済みに見せない（下書き/保存中/保存済み、推定/確定をラベルで区別する）。
- 中断・再開を許容する（部分入力の保持、自動保存、離脱前の警告、続きから再開できる導線）。
- 誤操作には回復導線を用意する（undo・削除の取り消し・直前状態への復元）。確認モーダルに頼り切らない（メタ下限 `M-RECOVERABILITY`）。
- 必須でない項目は後回しを許し、部分的な完了状態を正しく表現する。

## 設計すべき5状態（生成・検証の必須リスト）

理想状態（データが揃った正常表示）以外に、**最低この5状態を必ず設計する**。各状態で「何を見せるか」を一言で持っておく。

| 状態 | いつ | 何を見せるか（一言） |
|------|------|-------------------|
| 空 empty | データが0件 | 次の行動を促す invitation（何が表示される場所か＋最初の一手のボタン）。空白で放置しない。 |
| 読込中 loading | 取得・処理の待ち時間 | 処理中であることと推定時間（スケルトン／進捗）。無反応に見せない・レイアウトを飛ばさない。 |
| エラー error | 取得・操作の失敗 | 原因＋回復（何が起きたか＋再試行/問い合わせ導線）。生のエラーコードや空画面にしない。 |
| 権限不足 permission | 閲覧・操作の権限なし | 理由と要求先（なぜ見えないか＋誰に申請するか）。ボタンを黙って消すだけにしない。 |
| 部分・劣化 partial | 一部失敗・取得途中・機能低下 | 取得できた範囲＋欠損の明示（既知の情報は出し、欠けた部分を正直に示す）。全体を巻き添えで隠さない。 |

5状態のどれかが欠けている UI は「理想状態だけの設計」であり、運用に耐えない。

## 統合 — この章の使える規則（チェックリスト）

1. 各画面を 空 / 読込中 / エラー / 権限不足 / 部分・劣化 の5状態で設計・検証したか
2. 固定件数・固定文言に依存せず、大量・長文・0件で崩れないか
3. 崩れても意味を保つ（Graceful Degradation）フォールバックがあるか
4. 運用・管理 UI に 更新者・更新日時・履歴 を追える導線があるか
5. 操作の結果が同じ文脈で確認でき、別画面に飛ばして因果を切っていないか
6. エラーは原因＋回復導線をセットで出しているか（コード放置・空画面でないか）
7. 非同期・遅延・処理中を可視化し、無反応に見せていないか
8. 未確定・処理中・推定値を確定済みに見せていないか
9. 中断・再開・部分入力を許容し、誤操作に回復導線（undo・復元）があるか

衝突時の優先: 重大変化の可視性(6,7) と 回復可能性(9) はメタ下限。静けさより重大なエラー・遅延の可視を、確認モーダルより undo を優先する。理想表示の美しさのために例外状態の明示を犠牲にしない。
