# 02 — Information Structure（情報構造）

見た目の前に、情報をどう分類・階層化・可読化するか。
関心: **分類・階層・関係・密度・再利用**。決定順の段2。

各項目は `思想 → その思想を使うための理論 → 使える規則` の3層。

---

## Wurman — 情報は整理されて初めて理解できる

**思想**
「どの軸で分類するか」が「どう理解されるか」を決める。情報設計の目的は表示ではなく理解。

**その思想を使うための理論**
人は軸のない情報を理解できない。整理軸は無限ではなく **LATCH** の5つに収束する — Location（位置）/ Alphabet（五十音・ABC）/ Time（時系列）/ Category（カテゴリ）/ Hierarchy（重要度・大小）。1つのリストに複数軸が混ざると「何順か」が読めず理解が崩れる。だから軸を**1つに明示選択**し、画面にその軸を表示する。さらに「入口（どこから来る）と出口（どこへ進む）」を定義し、行き止まりを作らない。

**使える規則**
- 一覧・並び替え・グルーピングは LATCH から軸を1つ選ぶ。未ソートの生データを初期表示しない。
- 現在どの軸で並んでいるかを画面に出す（ソート中の列ヘッダをハイライト等）。
- 一覧→詳細→関連→検索の経路を用意し、詳細を dead end にしない。

## Legibility — 読める構造は操作できる構造

**思想**
複雑なシステムは、人間が読める形に変換されて初めて操作対象になる。可読性は美しさより前にある最低条件。

**その思想を使うための理論**
「読める」とは、現在地・全体像・関係・状態・次の行動が判別できること。都市の可読性（Lynch）の語彙で運用できる — Landmark（現在地の目印）/ Path（経路）/ District（まとまり）/ Edge（境界）。視覚評価の基準を「きれいか」ではなく「読めるか」に置き、衝突したら可読性を優先する（メタ下限 `M-LEGIBILITY`）。「目立つ」より「**識別できる**」を優先する。

**使える規則**
- すべての画面に現在地の手がかり（active 状態・固定ナビ・パンくず・セクション見出し）を置く。
- 意味的にまとまる要素を束ね、境界を 線・余白・背景差 のいずれかで明示する（曖昧な区切りを残さない）。
- 情報過多のときは、要素を減らすだけでなく要素間の関係（グルーピング・階層）を読めるようにする。

## Vignelli — デザインとは秩序を与えること

**思想**
グリッド・タイポグラフィ・制約に基づく一貫性が信頼を生む。情報の意味的順序を視覚構造へ翻訳する。

**その思想を使うための理論**
秩序は「選択肢を意図的に絞る」ことで生まれる。使う書体・色・間隔の種類を限定（トークン化）し、その場限りの中間値を作らない。視覚的な分類（サイズ・ウェイト・色）は情報の重要度と一致させる — 見た目の都合で重要度を逆転させない。例外表現（special case）はコストなので、増やす前に既存パターンで表現できないか検証する。

**使える規則**
- 要素はグリッドに配置し、余白・サイズはトークンから選ぶ（恣意的座標・中間値を作らない）。
- 最も目立つ要素 = 最も重要な情報、になっているか確認する。
- 同種の要素は同一表現に統一し、例外コンポーネントを最小化する。

## Tufte — 装飾を減らし、情報密度を上げる

**思想**
複雑な情報を高密度・明瞭・誠実に示す。情報を減らしすぎて判断材料を失うことを危険とみなす。

**その思想を使うための理論**
インクには2種ある — データを伝える **data-ink** と、伝えない **chartjunk**（背景塗り・立体化・過剰な枠・装飾）。後者を削り前者を残すと、信号対雑音比が上がり比較・変化・異常が読めるようになる。比較は **同一スケール・同一軸**（small multiples）でなければ成立しない。重要なのは「ミニマル ≠ 情報不足」。余白や抑制を理由に**判断材料（現在値・基準値・差分・比較対象）を削ってはならない**（メタ下限 `M-JUDGMENT-MATERIAL`）。正しく構造化されていれば高密度は許容される。

**使える規則**
- data-ink を最大化し chartjunk（3D・装飾背景・意味のない色分け・過剰な枠）を削る。
- 比較対象は同一スケール・同一軸・同一フォーマットで並べる。軸の起点・比率を歪めない。
- ミニマル化のとき「判断に必要な情報が欠けていないか」を必ず確認する。
- 状態表示は現在値だけでなく基準値・差分を併記する。

## Alexander — 良い設計には再利用できるパターンがある

**思想**
UI は単発の画面ではなく、`文脈 + 問題 + 解決` として繰り返し現れるパターンの集合体。

**その思想を使うための理論**
同じ問題に毎回違う解を作ると、体系が侵食され全体の質が落ちる。だから繰り返す構造をパターン/コンポーネントへ抽象化し、例外も one-off の見た目で処理せず既存パターンの拡張として組み込む。各パターンは「どの文脈で・どの問題を・どう解決するか」の三点で定義する（見た目だけのテンプレートにしない）。見た目が同じでも意味が違えば別パターン、意味が同じなら統一する。

**使える規則**
- 繰り返す構造は再利用可能なコンポーネントとして定義する。
- 例外を one-off カスタムで処理しない。既存パターンの拡張で表現する。
- コンポーネントは「文脈・問題・解決」を持たせる（意味で分け、意味で統一する）。

---

## 統合 — この章の使える規則（チェックリスト）

1. リストは単一の LATCH 軸で並び、その軸が画面に表示されているか
2. 現在地（active / breadcrumb / 見出し）が読めるか
3. セクション境界が線・余白・背景差で明示されているか
4. 最も目立つ要素が最も重要な情報か
5. data-ink が最大化され chartjunk が削られているか
6. 比較は同一スケール・同一軸か
7. ミニマル化で判断材料を削っていないか
8. 繰り返す構造がパターン化され、one-off がないか

衝突時の優先: 可読性(2,3) と 判断材料(7) はメタ下限。装飾削減(5)より優先する。
