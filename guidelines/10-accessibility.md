# 10 — Accessibility（アクセシビリティ）

造形を決めた後、それが「誰にとっても読めて操作できるか」を保証する。
関心: **コントラスト・ターゲット・焦点**。決定順の段6（抑制と耐性）。ここで扱うのは好みではなく**品質の下限（floor）**であり、満たさない設計は完成していない。

各項目は `思想 → その思想を使うための理論 → 使える規則` の3層。

---

## WCAG コントラスト — 読める下限を数値で保証する

**思想**
読めることは美しさより先に来る。可読性は主観ではなく、知覚の限界として数値で定義できる（WCAG）。

**その思想を使うための理論**
文字が読めるかは、文字色と背景色の**輝度比（コントラスト比）**で決まる。人間のコントラスト感度は加齢・低視力・屋外の照り返し・低品質ディスプレイで簡単に低下する。WCAG の `4.5:1` は、視力 0.5（20/40）程度まで低下した人でも本文を読める下限として導かれた値であり、デザイナーの正常な視覚で「読める」と感じる閾値より厳しい。大きな文字（18px以上 or 太字相当）は線が太く低コントラストでも字形を解像できるため `3:1` まで緩められる。アイコンやボーダー等の UI 部品も、形が読めなければ操作できないため `3:1` を要求する。重要なのは「自分が読めるか」ではなく「比率を満たすか」で判定すること。薄いグレー文字（`text-gray-400` 等）を本文に使うのは、この下限を主観で割る典型的な違反。

**使える規則**
- 本文テキストはコントラスト比 `4.5:1` 以上（`tokens.json` wcag.contrastNormalText）。
- 大きな文字（18px以上 or 太字）は `3:1` 以上（contrastLargeText）。
- アイコン・ボーダー・フォーム枠などの UI 部品は `3:1` 以上（contrastUIComponents）。
- 薄い文字色を本文に使わない。`text-black` も避け `text-slate-900` を使う。
- 比率は目視で判定せず、必ず数値で確認する。

## 色覚多様性 — 色は情報の唯一の担い手にしない

**思想**
色は速いが、全員に同じには届かない。状態の意味を色だけに載せると、一定数のユーザーには情報が消える。

**その思想を使うための理論**
男性の約8%（女性の約0.5%）が先天的な色覚特性を持ち、その多くが赤と緑の弁別が苦手（P型・D型）。成功＝緑／失敗＝赤のような「色だけ」の符号化は、この人たちにとって**同じ色**に見え、情報量がゼロになる。さらに補色（赤×緑、青×紫）を文字と背景で直接隣接させると、輝度差が小さい場合に滲み・ハレーションが起きて誰にとっても読みにくい。解決は**冗長符号化**——同じ意味を色に加えてアイコン・ラベル・テキスト・位置でも二重に伝える。こうすれば色が落ちても意味が残る。これは WCAG 1.4.1（色だけに依存しない）の要請そのもの。色は情報を**強める**補助に使い、唯一の担い手にはしない。

**使える規則**
- 状態（成功/失敗/警告）は色＋アイコン＋ラベルで二重化する（WCAG 1.4.1）。
- 補色（赤×緑、青×紫、黄×白）を文字×背景で直接使わない。明度差で区別する。
- リンク・必須項目・エラー箇所を色のみで示さず、下線・記号・テキストを併用する。
- セマンティックカラーは赤＝エラー/緑＝成功/黄＝警告に限定し、1画面の補色配色は1〜2箇所まで。

## Fitts / Apple — 押せる下限と届く場所

**思想**
操作できることは、見えることと同じくらい身体的な問題。ターゲットは指で正確に押せる大きさと、無理なく届く位置になければならない。

**その思想を使うための理論**
タップの所要時間と誤タップ率は Fitts の法則 `T = a + b·log₂(D/W + 1)` に従い、ターゲット幅 `W` が大きく距離 `D` が短いほど速く正確になる。人間の指の腹の接触面は 8〜14mm あり、これより小さい当たり判定は中心を外しやすい。Material の `48px`（約9mm）/ Apple の `44pt` はこの生理から導かれた下限で、主要操作ほど大きく取る。視覚サイズが小さいアイコンでも、透明パディングで**当たり判定**を 48px に広げれば見た目を崩さず操作性を確保できる。さらにスマホ大型化で片手時に親指が届く範囲（Natural Zone＝画面下部）は限られるため、主要 CTA は下部に置くと `D` が縮みエラーが減る。ベゼルレス端末では四隅がクリップされるので safe-area-inset で可視領域を保証する。

**使える規則**
- 主要操作（ナビ・FAB・完了）は最小 `48px × 48px`（`tokens.json` touchTargets.primary）。
- 二次操作（リスト項目・設定トグル）は最小 `44px × 44px` を下回らない（touchTargets.secondary）。
- ターゲット間隔は最小 `8px`（touchTargets.spacing）。
- アイコンボタンは視覚24px＋padding12pxで当たり判定48pxを形成する。
- モバイルの主要CTAは画面下部の親指ゾーンに置く。
- `env(safe-area-inset-*)` を必ず適用する。

## 焦点とキーボード — マウスなしで操作が完結する

**思想**
ポインタは唯一の入力手段ではない。キーボード・支援技術・運動制約のあるユーザーでも、迷わず操作を完結できなければならない。

**その思想を使うための理論**
マウスやタッチを使えない／使いにくいユーザーは、Tab で要素を移動し Enter/Space で実行する。このとき「いま自分がどこにいるか」を示す唯一の手がかりが**フォーカスリング**で、これを `outline:none` で消すと画面上の現在地が失われ、操作が不可能になる（見た目のために消すのは a11y 破壊の典型）。さらに操作経路がキーボードで辿れない（ホバーでしか開かない・クリック専用の div 等）と、そもそも到達できない要素が生まれる。曖昧なアイコンだけのボタンは、視覚的にも支援技術にも意味が伝わらないため、Apple HIG の Clarity に従いラベル（可視テキスト or `aria-label`）を併用する。前庭障害のあるユーザーには過剰な動きが不快・発作要因になり得るため、`prefers-reduced-motion` を尊重する（モーションの詳細は → 09）。

**使える規則**
- フォーカスを必ず可視にする。`outline:none` で消したら必ず代替の可視リング（`focus-visible` 等）を与える。
- すべての操作をキーボードだけで完結できるようにする（Tab で到達、Enter/Space で実行、論理的なフォーカス順序）。
- ホバー専用 UI を作らない。クリック要素はネイティブの `button`/`a` を使い、キーボード操作を担保する。
- 曖昧なアイコンにはラベル or `aria-label` を併用する（Apple HIG: Clarity、→ 03 と連動）。
- `prefers-reduced-motion: reduce` を尊重し、必須でないアニメーションを止める（→ 09 motion）。

## 統合 — この章の使える規則（チェックリスト）

以下はすべて**品質の下限（floor）**であり、好みで割ってよい項目はない。1つでも欠ければ「一部のユーザーには使えない UI」になる。

1. 本文テキストのコントラスト比は `4.5:1` 以上か（大文字・太字は `3:1`、UI部品は `3:1`）
2. 薄い文字色・`text-black` を本文に使っていないか
3. 状態（成功/失敗/警告）を色＋アイコン＋ラベルで二重化しているか（色だけにしていないか）
4. 補色を文字×背景で直接使っていないか／1画面の補色配色は1〜2箇所までか
5. 主要操作は `48px`、二次操作は `44px`、間隔は `8px` を満たすか
6. アイコンボタンの当たり判定は視覚サイズと分離して48pxあるか
7. モバイル主要CTAは下部の親指ゾーンにあり、`safe-area-inset` を適用しているか
8. フォーカスが必ず可視か（`outline:none` を代替なしで放置していないか）
9. キーボードだけで操作が完結するか／曖昧アイコンにラベルがあるか
10. `prefers-reduced-motion` を尊重しているか（→ 09）

衝突時の優先: コントラスト(1)・色非依存(3)・フォーカス可視(8)・キーボード完結(9) は譲れない最下限。造形上の都合（薄い色・装飾的な無枠フォーカス）より a11y を常に優先する。
