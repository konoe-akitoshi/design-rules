# 06 — Typography（タイポグラフィ）

情報構造で決めた重要度と読み順を、文字組みで視覚に翻訳する。
関心: **階層・読み順**。決定順の段5（造形）。色や角丸より先に、文字のサイズ・ウェイト・行間・余白で「どこから読むか」を確定させる。

各項目は `思想 → その思想を使うための理論 → 使える規則` の3層。
値は `tokens.json`（spacing で縦リズム）を参照。本章の type scale 数値は**参照デフォルト**で、プロダクトの基準フォントに合わせて等比でずらしてよい。

---

## International Typographic Style（Swiss）— タイポグラフィは構造化の手段

**思想**
タイポグラフィは装飾ではなく、情報の流れと階層を視覚化する手段。サイズ・ウェイト・行間・余白で階層を作り、サンセリフ基調で客観性（伝達優先）を保つ。

**その思想を使うための理論**
読み手は文字を読む前に、まず**面として**画面を見る。そのとき効くのが「コントラスト＝差」。サイズ差・ウェイト差・余白差が大きい要素ほど先に目に入り、差がなければ並列＝同じ重要度と読まれる。つまり階層は色や罫線で「飾る」ものではなく、**4つの変数（サイズ／ウェイト／行間／余白）の差**で組み上げるもの。Swiss が装飾を削るのは禁欲ではなく、装飾が入るとこの差が濁って階層が読めなくなるから。フォントは1〜2ファミリに絞り、書体の違いではなく**同一ファミリ内のサイズ・ウェイトの差**で語る。これにより「見た目の差＝意味の差」という対応が崩れない。

**使える規則**
- 階層はサイズ・ウェイト・行間・余白の差で作る。装飾（罫線・影・色面）で階層を代替しない。
- **道具/アプリ（B）**: 本文用1＋数値/コード用 mono の最大2系統に絞り、サンセリフ基調・可読性最優先。見出し専用書体を増やさない。
- **ブランド/編集（A）**: identity を担う**ディスプレイ書体を1つ選んでよい**（編集的ならセリフ、技術的ならグロテスク/モノ）＋噛み合う本文1書体。Web フォント可（→ [01「題材からブランドを導く」](./01-foundations.md)）。系統は最大2＋必要なら mono。「見出し専用書体を増やさない」はBの規則で、Aには当てない。ただし**選んだ書体は実際に使う**——1ファミリで size/weight/tracking が階層を担えるなら、気分で display/mono を足さない（未使用ファミリは identity をむしろ濁す）。
- 隣り合う階層には**知覚できる差**を付ける（サイズ最低約1.2倍 or 明確なウェイト差）。差が小さいなら階層を分けない。

### ディスプレイ/見出しスケール（参照）
本章の本文寄りスケールはアプリ基準。**ブランドのヒーロー見出しは大きく振る** — 例 `clamp(40px, 6vw, 88px)`、line-height 0.95–1.05（タイト）、letter-spacing -0.02em。本文 16–18px。サイズ差は重要度差に対応させ、装飾目的の巨大化はしない。

## Vignelli（秩序）— スケールは限定値の集合から選ぶ

**思想**
デザインは秩序を与える行為。タイプスケールは無限の自由ではなく、**限定された値の集合（トークン）**から選ぶ。視覚上の分類は、意味的な順序と一致していなければならない。

**その思想を使うための理論**
中間値を許すと階層は無限に増殖する（15px と 16px と 17px が混在する）。差が小さい階層は読み手に**区別の意図があるのか偶然なのか**を判断させ、毎回コストを払わせる。だから値を少数の段に固定し、新しい強調が必要になっても**段を増やさず既存の段に割り当てる**。これは制約ではなく圧縮で、段が少ないほど1段あたりの意味が濃くなり、読み手は「大きい＝重要」を学習一回で全画面に適用できる（`M-CONSISTENCY`）。重要なのは並び順の不変条件 — **size順 = weight順 = 意味の重要度順**を常に一致させること。見出しが本文より小さい、補足が本文より太い、といった逆転は秩序を壊し、視覚分類を信用できなくする。

**使える規則**
- type scale は下表の限定段のみを使う。中間サイズを新規に作らない。
- ウェイトも限定する（例: 400 / 600 の2値、必要時のみ 500 を追加）。太字の乱用で段を曖昧にしない。
- 新しい強調が必要でも段を増やさず、既存段（サイズ＋ウェイト＋色の組）に割り当てる。
- 視覚の順序と意味の順序を一致させる（重要なものほど大きく/太く、補足ほど小さく/淡く。逆転禁止）。

### 参照タイプスケール（限定段・基準16px系）

| 役割 | size | weight | line-height | 用途 |
|------|------|--------|-------------|------|
| H1 / ページタイトル | 28px | 600 | 1.25 | 画面に1つ。最上位 |
| H2 / セクション見出し | 22px | 600 | 1.3 | セクションの区切り |
| H3 / サブ見出し | 18px | 600 | 1.4 | グループ・カード見出し |
| Body / 本文 | 14px | 400 | 1.6 | 既定のテキスト |
| Label / 強調ラベル | 14px | 600 | 1.4 | フォーム項目名・強調語 |
| Caption / 補足 | 12px | 400 | 1.5 | 注記・メタ情報（色を淡く） |
| Overline / 区分ラベル | 12px | 600 | 1.4 | 大文字＋字間。小見出し代替 |
| Mono / 数値・コード | 13px | 400 | 1.5 | 等幅。桁揃えが要る値 |

数値は参照デフォルト。基準を変えるときも段数は増やさず、等比（約1.2〜1.25）で全段をずらす。

## 読み順とリーダビリティ — 階層で読み順を一意にする

**思想**
良い文字組みは、最重要 → 次点 → 補足という**唯一の読み順**を視覚だけで決めてしまう。読み手は「どこから読むか」を迷わない。

**その思想を使うための理論**
階層がなぜ読むコストを下げるか — 人は全体を平等に走査せず、最もコントラストの強い要素から読み始め、そこを起点に視線を運ぶ。階層が明確なら走査は1パスで済む。階層が平坦だと、読み手はどれが幹でどれが枝か分からず**再走査**を強いられ、コストが跳ね上がる。だから1画面の最上位は1つに絞り（H1 が複数あると起点が割れる）、同位の要素は同じ段で揃える。次に効くのが**行長（measure）と行間**。1行が長すぎると行頭から次行の行頭へ視線が戻れず行を見失い、短すぎると改行が増えてリズムが切れる。行間（line-height）は行の塊を「段落」という1ブロックに見せる接着剤で、狭いと行が混じり、広いと行同士の所属が切れる。本文・見出し・補足のサイズとウェイトは、**意味（幹・枝・葉）と結びつけて**初めて機能する — 装飾目的でサイズを上げ下げすると、この意味対応が壊れて読み順が嘘になる。

**使える規則**
- 1画面の最上位見出し（H1）は1つ。同じ重要度の要素は同じ段で揃える。
- 本文の行長は**欧文45〜75字／和文40〜50字**に収める（`max-width` 目安 36〜42em）。広い画面でも本文を全幅に流さない。
- 行間は役割で変える: 本文 1.5〜1.65、見出し 1.2〜1.35、密な数値リスト 1.4 前後。
- サイズ・ウェイトは意味と結びつけて使う。**装飾目的のサイズ変更を禁止**（目立たせたいだけなら段ではなく色/余白で対応）。
- 補足は小さく＋淡く（色のコントラストを下げる）。ただし WCAG 下限は守る（本文 4.5:1／大きな文字 3:1）。

### 縦リズム（spacing トークンで段落・見出しの間隔を揃える）

ブロック間の余白も限定値（`tokens.json` の spacing）から取り、4px グリッドに乗せる。

- 見出しと直後の本文は近く（`spacing-2` 8px / `spacing-3` 12px）、見出しと**前の**ブロックは遠く（`spacing-8` 32px）。近接で「見出しは下の内容に属する」を示す。
- 段落間は `spacing-4`（16px）。セクション間は `spacing-8`（32px）〜`spacing-12`（48px）。
- 行送り・ブロック間は 4 の倍数に揃え、リズムの拍を一定にする（半端な 17px・23px を作らない）。

## 統合 — この章の使える規則（チェックリスト）

1. 階層をサイズ・ウェイト・行間・余白の差で作っているか（装飾で代替していないか）
2. フォントは本文1＋mono の最大2系統に絞れているか
3. type scale は限定段のみか（中間サイズを新設していないか）
4. ウェイトは2〜3値に限定されているか（太字の乱用がないか）
5. size順 = weight順 = 意味の重要度順が一致しているか（逆転がないか）
6. 1画面の H1 は1つか／同位要素は同じ段で揃っているか
7. 本文の行長が 45〜75字（和文40〜50字）に収まっているか
8. 行間が役割で適切か（本文 1.5〜1.65／見出し 1.2〜1.35）
9. 装飾目的のサイズ変更をしていないか（目立たせは色/余白で）
10. 段落・見出しの間隔が spacing トークン（4の倍数）に乗っているか
11. 補足は小さく淡くしつつ WCAG 下限（4.5:1／3:1）を守っているか

衝突時の優先: 一貫性（3・5）と可読性下限（7・8・11）はメタ下限。見栄えのための段追加・行長超過・コントラスト低下より、限定スケールと読みやすさを優先する。
