# 03 — Interaction & Feedback（操作とフィードバック）

整理された情報を、人間が自然に操作し結果を受け取れる体験へ接続する。
関心: **認知・操作・フィードバック・注意**。決定順の段3。情報があることと、操作・理解できることは別問題。

> **層**: Norman（認知）/ Calm（注意）/ Apple HIG は **A** — 操作対象（リンク・ボタン等）があれば静的ページにも効く。
> Cybernetics（観測→判断→制御→FB）/ Raskin（操作コスト）は **B** — 動的な状態・反復操作があるときだけ。ブランドページには Cybernetics を当てない。

各項目は `思想 → その思想を使うための理論 → 使える規則` の3層。

---

## Cybernetics — 画面は制御ループの一部

**思想**
UI は静的な表示面ではなく、観測→判断→制御→フィードバックの制御ループの一部。

**その思想を使うための理論**
動的なシステム（監視・AI Agent・ロボット・運用）でユーザーがやっているのは「状態を見て、目標との差を判断し、介入し、結果を確認する」循環。この4要素のどれか1つでも欠けるとループが閉じず、ユーザーは次の手を打てない。だから状態を扱う画面には4要素を必ず揃え、**操作→結果を同じ文脈で閉じる**（別画面に飛んで因果が切れない）。フィードバックは「何が起きたか」と「次に何をすべきか」の両方を含める。

**使える規則**
- 状態を扱うUIに 観測（現在状態）/ 判断（目標・基準との差）/ 制御（操作手段）/ フィードバック（結果）を揃える。
- 操作の結果は、操作した場所から目を離さずに確認できるようにする。
- エラーには検出（何が違うか）と復旧導線（どう直すか）をセットで出す。

## Norman — 見える手がかりだけから理解される

**思想**
ユーザーは設計者の意図ではなく「見える手がかり」から理解・操作する。エラーはユーザーの失敗ではなく設計の問題。

**その思想を使うための理論**
操作の成否は4つで決まる — **Affordance/Signifier**（何ができるか見える）/ **Mapping**（操作と結果の対応が自然）/ **Feedback**（何が起きたか返る）/ **Conceptual Model**（画面構造がユーザーの頭の中と一致）。整理された構造 ≠ 理解しやすい構造。検証基準は「設計者に分かる」ではなく「ユーザーがそう感じるか」。エラーは責めずに、原因・回復・次の一手を示す。さらに入力制約を**事前に**伝えればエラーの発生自体を防げる。

**使える規則**
- 操作可能な要素に signifier（押せる見た目・入力枠・ホバー・カーソル変化）を付ける。
- すべての操作に即時フィードバックを返す（押下・送信中・完了）。
- コントロールは対象の近くに置き、並び・向きを結果と一致させる（自然な mapping）。
- エラー文は原因＋回復策を示し、責めない（「不正な値」→「日付は YYYY-MM-DD で」）。
- 入力制約はプレースホルダ・補助テキスト・例で事前に伝える。

## Raskin — UI は時間の中で繰り返される作業の流れ

**思想**
UI は「見た目」ではなく「繰り返される作業」。操作コスト・認知負荷・作業速度を優先する。

**その思想を使うための理論**
熟練者の速度を支配するのは アクセスコスト（クリック数・移動距離・階層の深さ）と モードエラー（同じ操作が文脈で別の意味を持つ混乱）。だから頻出操作は最短化し、キーボード経路を併設する。モードは極力なくし、不可避なら現在モードを常時表示する。記憶を要求せず画面上で**認識**させる（recognition over recall）。そして破壊的操作を毎回の確認モーダルで止めない — **undo を優先**し、確認は本当に不可逆な操作だけに限定する（メタ下限 `M-RECOVERABILITY`）。

**使える規則**
- 頻出操作はクリック数・深さを最小化し、主要操作にショートカット/コマンドパレットを用意する。
- モードを最小化し、不可避なら現在モードを常時表示する。
- 前画面の情報を覚えさせない（必要な値・選択肢・直前入力を画面に出す）。
- 破壊的操作は undo を提供し、確認モーダルで熟練者の流れを止めない。
- **URL に状態を反映する**: フィルタ・タブ・ページネーション・展開パネルはディープリンク可能にする（リロード・共有・Back/Forward で作業状態が失われるのは「前画面を覚えさせる」のと同じ記憶の強要）。Back/Forward でスクロール位置も復元する。

## Calm Technology — 必要な時だけ注意を引く

**思想**
技術は通常時は静かに周辺にあり、必要な時だけ前景化する。注意は有限資源。

**その思想を使うための理論**
注意には予算がある。すべてを強調すると優先順位が消え（全部太字＝何も目立たない）、本当に重要な変化が埋もれる。だから情報を「中心（今すぐ注意が必要）」と「周辺（状態として背景提示）」に分け、平常は控えめに、変化のあった対象だけを強調する。通知は重要度で出し分け、割り込み（モーダル/音）は緊急かつ行動が必要なものに限定する。ただし「静か ≠ 不明瞭」。静けさを理由に重大なエラー・期限を埋もれさせてはならない（メタ下限 `M-CRITICAL-VISIBILITY`）。

**使える規則**
- 注意の階層を設計する。平常状態を過剰に目立たせない（常時アニメ・常時赤バッジを避ける）。
- 強調は変化のあった対象だけに使い、1画面のアクセント・割り込みを絞る。
- 割り込み通知は緊急時のみ。通常更新は控えめなインジケータ・ambient 表示に留める。
- それでも重大な異常・エラー・期限は確実に視認できる明確さを確保する。

## Apple HIG — 明瞭さ・従属・自然なフィードバック

**思想**
明瞭さ（Clarity）・内容への従属（Deference）・関係を伝える深さ（Depth）・即時で自然なフィードバック。見た目の模倣ではなく直感性が本質。

**その思想を使うための理論**
ユーザーは画面を説明書として読まず、見た瞬間に手がかりを得る。だから文字の可読性とアイコンの意味の明確さを美しさより優先し、曖昧なアイコンにはラベルを併用する（Clarity）。UI のクローム（枠・装飾）は抑え、作業対象を主役にする（Deference、Rams と連動）。立体感・遷移は装飾でなく「どこから来てどこへ行くか」を伝えるためだけに使う（機能的 Depth）。フィードバックは遅すぎ（無反応に見える）も過剰（派手・長尺）も避け、自然な速度・量にする。プラットフォーム慣習（戻る・ナビ・標準ジェスチャ）は理由なく破らない。

**使える規則**
- 曖昧なアイコンにはラベルを併用する（Clarity）。
- UI クロームを抑え、コンテンツを最も目立たせる（Deference）。
- 影・立体・遷移は前後関係・階層を伝えるときだけ使う（装飾的な影を禁止）。
- フィードバックは自然な速度・量に。標準操作（戻る・ナビ）を期待通りに動かす。

## 文言（マイクロコピー）— 操作の語彙も UI の一部

**思想**
ボタンの文字・エラー文・トーストは装飾でなく操作の一部。曖昧な語はユーザーに「押すと何が起きるか」の推測を強いる（Norman の Conceptual Model）。

**使える規則**
- アクションは**動詞＋対象**（「Deploy Project」「変更を保存」）。`Confirm` / `OK` / `Submit` / 裸の動詞だけ、は禁止（何を確定するのか読めない）。
- **語彙を往復で一貫させる**: ボタン「公開」→トースト「公開しました」。操作と結果で別の語を使わない。
- 進行中は現在進行形＋省略記号（「Deploying…」「保存中…」）。完了トーストに "successfully" 系の冗語を足さない（対象と結果だけ言う）。
- エラーは謝罪でなく**原因＋修正方法**（「不正な値です」→「日付は YYYY-MM-DD 形式で入力してください」）。
- 能動態で書く。ラベル・見出し・ボタンに文末ピリオド（。）を付けない。

## 統合 — この章の使える規則（チェックリスト）

1. 状態UIに 観測・判断・制御・フィードバック が揃っているか
2. 操作の結果が同じ文脈で確認できるか
3. 操作可能な要素に signifier があるか／全操作に即時フィードバックがあるか
4. エラーは原因＋回復策を示し、責めていないか
5. 破壊的操作に undo があるか（毎回の確認モーダルで止めていないか）
6. 平常は静かで、強調が変化のあった対象に絞られているか
7. それでも重大な異常・期限は確実に視認できるか
8. 曖昧アイコンにラベルがあるか／装飾的な影を使っていないか
9. フィルタ・タブ等の状態が URL に反映されているか（※B。ディープリンク・Back/Forward 復元）
10. 文言が「動詞＋対象」で、操作→結果の語彙が一貫しているか（`OK`/`Confirm` だけのボタンが無いか）

衝突時の優先: 回復可能性(5) と 重大変化の可視性(7) はメタ下限。確認より undo、静けさより重大変化の可視を優先。
