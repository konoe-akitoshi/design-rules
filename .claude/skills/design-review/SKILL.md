# Design Review — 検査（直さない）

指定された UI をガイドラインに照らして検査し、**修正せず**根拠付きで報告する。直してほしい場合は `/design-fix`。

## 読み込む（必須）

1. `${CLAUDE_SKILL_DIR}/../../../prohibited.md` — 禁止パターン・AI収束スタイル・実装床
2. `${CLAUDE_SKILL_DIR}/../../../tokens.json` — トークン定義
3. `${CLAUDE_SKILL_DIR}/../../../guidelines/README.md` — 章構成と「優先順位」（floor: 可読性/判断材料/回復可能性/重大変化の可視性/実用性）

原則検査では該当章の**章末「統合チェックリスト」**を基準にする（02/03/07/08/10 は最低限読む）。

## 検査対象

$ARGUMENTS

## 検査手順

### 1. 機械的検出（先に回す）
対象がHTMLなら lint を実行し、結果をそのまま検査結果に取り込む:
```
python ${CLAUDE_SKILL_DIR}/../../../tools/design-lint.py <file>
```
JSX 等 lint 対象外の場合は同じ観点を目視で当てる: 禁止パターン（shadow-lg / text-black / カラーバー / グラデ多用 / 絵文字）・トークン外の値・対称パディング・4の倍数外 spacing・角丸階層・`transition: all`・入力16px未満・img寸法。

### 2. 原則レベル検査（章末チェックリストを当てる）
- **02 情報構造**: 整理軸が単一か / 現在地が読めるか / 重要度と視覚階層が一致か / 判断材料を削っていないか
- **03 操作とFB**: 状態UIに観測・判断・制御・FBが揃うか / signifier・即時FBがあるか / エラーは原因＋回復か / 破壊操作に undo か / 文言は動詞＋対象か
- **07 抑制と造形**: クロームが内容より目立たないか / 根拠なき造形が無いか / 削れる装飾を削ったか
- **08 状態と耐性**: 空/読込/エラー/権限/部分の5状態があるか / 大量・長文で崩れないか
- **10 a11y**: コントラスト / 色のみ依存 / タッチターゲット / focus可視 / アイコンラベル
- **AI収束スタイル**（prohibited）: 3類型にデフォルトで落ちていないか。落ちている場合、題材からの導出理由があるか

## 出力フォーマット

```
## Design Review 結果

### サマリー
- Critical（floor違反・禁止パターン・WCAG）: X件
- Warning（原則違反・トークン逸脱）: Y件
- Info（改善提案）: Z件
（lint: N error / M warn — 実行した場合）

### 詳細
#### [Critical] <違反名>（章 or ルール名）
- 箇所: ファイル:行
- なぜ問題か: <理論を1〜2文>
- 修正案: <具体的に>
```

重大度: floor 違反（可読性・判断材料・回復・重大変化の可視・a11y）と禁止パターンを Critical、原則違反・トークン逸脱を Warning、提案を Info とする。指摘には必ず根拠（章 or lint ルール名）を添える。
