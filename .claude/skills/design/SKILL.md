# Design — 要件から実装まで

デザインガイドライン準拠で UI を設計・生成する。要件の確認 → 戦略 → 生成 → セルフチェックを1つの流れで行う。

## 読み込む（必須）

1. `${CLAUDE_SKILL_DIR}/../../../CLAUDE.md` — クイックリファレンス（まずこれ）
2. `${CLAUDE_SKILL_DIR}/../../../guidelines/README.md` — 決定順・A/B判別・優先順位
3. `${CLAUDE_SKILL_DIR}/../../../tokens.json` — トークン定義（値の正）
4. `${CLAUDE_SKILL_DIR}/../../../guidelines/values/color-system.md` — 配色の実値・状態色の導出・ダーク導出
5. `${CLAUDE_SKILL_DIR}/../../../prohibited.md` — 禁止パターン＋AI収束スタイル＋実装床（生成後のセルフチェックに使う）

タスクに応じて該当章を追加で読む: レイアウト→04 / 配色→05 / タイポ→06 / 抑制→07 / 状態UI→03・08 / モーション→09 / 部品の作り→components/。

## プロセス（この順で。見た目から始めない）

### 0. 要件の確認（曖昧なときだけ）
「誰が・何のために・この画面の唯一の仕事は何か」が1文で言えるかを確認する。
- 言える → すぐ次へ（質問で待たせない）。
- 言えない → **最大2〜3問だけ**質問する: ①何のシステムで誰が使うか ②ユーザーは何を観測・判断・制御するか ③失敗・空・権限不足のとき何が起きるべきか。仮定して進む場合は仮定を明示する。

### 1. A/B判別と戦略
- **ブランドサイト・記事・LP** → A章のみ。01「題材から構造を導く・汎用テンプレ骨格禁止」を必ず通す（eyebrow＋大見出し＋2ボタン＋右図版、等分3カラム、中央引用、装飾連番を既定にしない）。
- **ダッシュボード・管理・運用ツール（動的な状態/操作がある）** → A＋B。扱う状態ごとに **観測→判断→制御→フィードバック** の4要素を1行ずつ言語化してから作る。ユーザーに暗算・記憶・画面往復をさせない（並置・事前計算・状態保持で肩代わり）。不可逆操作の human-in-the-loop 点を特定する。

### 2. 生成（決定順: 目的→構造→操作→レイアウト→造形→抑制→検証）
- 値は tokens.json / color-system の値のみ（マジックナンバー禁止）。派生色（hover 等）は手置きせず `color-mix`/`oklch(from …)` で導出。
- 繰り返す構造はパターン（コンポーネント）にする: 「文脈＋問題＋解決」で定義し、**6マイクロステート**（default/hover/focus/active/disabled/loading）と**5例外状態**（空/読込/エラー/権限/部分）を最初から内包。状態は variant/props で表現。
- 実装床（prohibited「実装・インタラクション」）: `<form>`+Enter送信・入力16px以上・`min-width:0`・img寸法明示・`<a href>`ナビ・transform/opacity のみアニメ・`transition: all` 禁止。

### 3. セルフチェック（出力前に必ず）
1. prohibited.md の禁止パターン・実装床に該当箇所がないか
2. **AI収束スタイル3類型**（クリーム紙×セリフ×テラコッタ／黒地×アシッド1色／hairline新聞風）にデフォルトで落ちていないか — 使うなら題材からの導出理由を言えること
3. トークン値と一致しているか／非対称パディングか／入れ子角丸 R_inner = R_outer − padding か
4. 見出し `text-wrap: balance`・本文 `pretty`・数値列 `tabular-nums`・本文行長 66ch 以内か
5. コントラスト（本文4.5:1／UI3:1）・focus可視・色のみ依存なし・タッチターゲット（主要48/二次44/間隔8）か

違反があれば修正してから出力する。機械検査したい場合は `python ${CLAUDE_SKILL_DIR}/../../../tools/design-lint.py <file>` が使える（HTML対象）。
